# Day 4 – Modules, Packages & Project Architecture (Part 2)

# Project Layering

One of the biggest differences between beginner and production code is **layering**.

Beginners usually write

```
UI

↓

Database

↓

Done
```

Everything is connected directly.

Professional applications separate responsibilities.

```
UI

↓

API

↓

Service

↓

Repository

↓

Database
```

Each layer has one responsibility.

---

# Why Layer Applications?

Suppose tomorrow you change your database from SQLite to PostgreSQL.

Without layers,

every file needs modification.

With layers,

only the repository changes.

The rest of the application remains untouched.

---

# Typical Project Layers

## Presentation Layer

Responsible for interacting with users.

Examples

- CLI
- React Frontend
- REST API
- Mobile App

Responsibilities

- Receive requests
- Display responses

Should NOT contain business logic.

---

## Service Layer

Contains business rules.

Example

```python
calculate_salary()

generate_invoice()

approve_leave()

calculate_discount()
```

The service layer answers

> "How should the application behave?"

---

## Repository Layer

Responsible for database communication.

Example

```python
save_student()

find_student()

delete_student()
```

Only this layer knows SQL.

---

## Database Layer

Stores data.

Examples

- SQLite
- PostgreSQL
- MySQL
- MongoDB

---

# Example Flow

Suppose a user registers.

```
React

↓

POST /register

↓

UserService.register()

↓

UserRepository.save()

↓

Database
```

Each layer performs only its own job.

---

# Why Not Skip Layers?

Bad

```python
@app.post("/register")

def register():

    cursor.execute(...)

    send_email()

    calculate_discount()

    print(...)
```

Everything is mixed together.

Hard to test.

Hard to maintain.

---

Better

```python
@app.post("/register")
```

↓

```python
UserService.register()
```

↓

```python
UserRepository.save()
```

↓

Database

Simple.

---

# Business Logic

Business logic is the heart of the application.

Examples

```
Attendance must be ≥75%

Tax = 18%

Password length ≥8

Maximum loan amount = ₹10,00,000
```

These rules belong in the Service Layer.

Not in

- UI
- Database
- API

---

# Utility Modules

Utilities contain reusable helper functions.

Example

```
utils/

    date.py

    email.py

    string.py

    validators.py
```

Good examples

```python
format_date()

validate_email()

slugify()

hash_password()
```

---

Avoid creating

```
utils.py
```

with 500 unrelated functions.

Split helpers logically.

---

# Configuration

Never hardcode values.

Bad

```python
API_KEY = "abc123"

DATABASE = "localhost"
```

Better

```
.env
```

```
DATABASE_URL

SECRET_KEY

EMAIL_PASSWORD
```

Read them in code.

Example

```python
import os

os.getenv("DATABASE_URL")
```

---

# Constants

Magic values reduce readability.

Bad

```python
if marks >= 35:
```

Why 35?

Better

```python
PASS_MARK = 35

if marks >= PASS_MARK:
```

Now the meaning is obvious.

---

# Avoid Code Duplication

Bad

```python
validate_email()

validate_phone()

validate_name()
```

Each repeats

```python
strip()

lower()

remove spaces
```

Extract common logic.

---

# Logging vs Printing

Beginners

```python
print("User Logged In")
```

Professional code

```python
logger.info("User Logged In")
```

Why?

Logging

- Can be saved to files
- Supports log levels
- Helps debugging
- Doesn't clutter user output

---

# Dependency Direction

Higher layers depend on lower layers.

```
Frontend

↓

API

↓

Service

↓

Repository

↓

Database
```

Lower layers should NOT depend on higher layers.

Wrong

```
Database

↓

Frontend
```

This creates tight coupling.

# Frappe Architecture

A Frappe application follows the same layering ideas.

Example

```
my_app/

    api.py

    hooks.py

    config/

    doctype/

        student/

            student.py

            student.json

            student.js

    utils/

    public/
```

---

# Understanding Frappe Structure

## api.py

Contains functions exposed to the frontend.

Example

```python
@frappe.whitelist()
def get_student():
    ...
```

---

## hooks.py

Framework configuration.

Examples

- Scheduled jobs
- Event hooks
- Permissions
- Installed apps

---

## doctype/

Contains business objects.

Example

```
Sales Order

Employee

Invoice

Student
```

Each DocType has

- Model
- Controller
- Metadata

---

## Controller

Example

```python
class Student(Document):

    def validate(self):
        ...
```

Business rules belong here.

---

# Where Should Business Logic Live?

Bad

```
API

↓

Everything
```

Good

```
API

↓

Controller

↓

Service

↓

Repository
```

Thin APIs.

Fat business logic.

# Fat Models, Thin Routes

Bad

```python
@app.post("/student")

def create():

    validate()

    save()

    send_email()

    generate_report()

    update_dashboard()

    ...
```

Huge route.

---

Better

```python
@app.post("/student")

StudentService.create_student()
```

All logic stays inside the service.

# Interview Questions

### What is project layering?

Separating an application into independent layers with distinct responsibilities.

---

### Where should business logic live?

Service layer (or controller layer in some frameworks like Frappe).

---

### Why use utilities?

To avoid duplicate helper functions.

---

### Why avoid hardcoded values?

They are difficult to maintain and change.

---

### Difference between logging and printing?

`print()`

- Temporary
- User output

Logging

- Persistent
- Debugging
- Production monitoring

---

### What belongs in `hooks.py`?

Framework configuration like events, scheduled tasks, permissions, and application hooks.

---

# Common Beginner Mistakes

❌ Writing SQL inside route handlers.

❌ Putting all logic inside API endpoints.

❌ Using one huge `utils.py`.

❌ Hardcoding passwords and API keys.

❌ Using `print()` for debugging in production.

❌ Mixing business logic with UI.

---

# LLM Code Review Checklist

Whenever ChatGPT generates a project structure, ask:

- Is the application layered?
- Is business logic separated from the API?
- Is database access isolated?
- Are utilities grouped logically?
- Are configuration values externalized?
- Is logging used instead of print?
- Are responsibilities clearly divided?

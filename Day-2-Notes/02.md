# Day 2 – Functions & Abstraction (Part 2)

# Default Parameters

Sometimes a parameter has a common value.

Instead of requiring the caller to provide it every time, give it a default.

Example

```python
def greet(name, message="Hello"):
    return f"{message}, {name}"
```

Usage

```python
greet("Alice")
```

Output

```
Hello, Alice
```

Override the default

```python
greet("Alice", "Welcome")
```

Output

```
Welcome, Alice
```

---

## Rules

Default parameters must come after required parameters.

Correct

```python
def add(a, b=0):
    return a + b
```

Wrong

```python
def add(a=0, b):
```

SyntaxError

---

## Mutable Default Argument (Important Interview Question)

Never use mutable objects as default values.

Bad

```python
def add_student(name, students=[]):
    students.append(name)
    return students
```

```python
print(add_student("Alice"))
print(add_student("Bob"))
```

Output

```
['Alice']
['Alice', 'Bob']
```

Why?

The list is created **only once** when the function is defined.

Every call reuses the same list.

---

Correct

```python
def add_student(name, students=None):
    if students is None:
        students = []

    students.append(name)
    return students
```

Each call gets a new list.

---

# Positional Arguments

Arguments are matched based on position.

```python
def introduce(name, age):
    print(name, age)

introduce("Alice", 20)
```

```
Alice 20
```

---

Wrong order

```python
introduce(20, "Alice")
```

```
20 Alice
```

Python doesn't know you swapped them.

---

# Keyword Arguments

Arguments can be passed using parameter names.

```python
introduce(age=20, name="Alice")
```

Output

```
Alice 20
```

Benefits

- Clearer
- Order doesn't matter
- Easier to read

---

# Mixing Positional and Keyword Arguments

Allowed

```python
introduce("Alice", age=20)
```

Not allowed

```python
introduce(name="Alice", 20)
```

Because positional arguments cannot come after keyword arguments.

---

# Variable-Length Arguments (`*args`)

Sometimes you don't know how many arguments the caller will provide.

```python
def total(*numbers):
    return sum(numbers)
```

Usage

```python
total(1,2,3)
```

```
6
```

```python
total(5,10,15,20)
```

```
50
```

---

## What is `args`?

Inside the function,

```python
numbers
```

is a tuple.

```python
def demo(*args):
    print(type(args))
```

Output

```
<class 'tuple'>
```

---

# Variable Keyword Arguments (`**kwargs`)

Accept any number of keyword arguments.

```python
def display(**details):
    print(details)
```

Usage

```python
display(name="Alice", age=20)
```

Output

```python
{
    "name":"Alice",
    "age":20
}
```

---

`kwargs` is a dictionary.

```python
print(type(details))
```

Output

```
<class 'dict'>
```

---

# When to Use `*args` and `**kwargs`

Good examples

- Logging
- Wrappers
- Decorators
- Flexible APIs

Avoid using them everywhere.

If the number of parameters is fixed, define them explicitly.

# Function Type Hints

Python is dynamically typed.

Type hints improve readability and tooling.

Example

```python
def add(a: int, b: int) -> int:
    return a + b
```

Meaning

```
a -> int

b -> int

returns -> int
```

---

## More Examples

String

```python
def greet(name: str) -> str:
    return f"Hello {name}"
```

Boolean

```python
def is_even(n: int) -> bool:
    return n % 2 == 0
```

List

```python
def average(scores: list[int]) -> float:
    return sum(scores) / len(scores)
```

Dictionary

```python
def student() -> dict[str, int]:
    ...
```

Optional

```python
from typing import Optional

def login(user: Optional[str]):
    ...
```

Meaning

```
str

or

None
```

---

# Do Type Hints Enforce Types?

No.

Example

```python
def add(a: int, b: int):
    return a + b

add("2", "3")
```

Python still runs.

Output

```
23
```

Type hints are mainly for:

- Developers
- IDEs
- Static analysis tools like mypy

---

# Why Type Hints Matter

Without hints

```python
def calculate(a, b):
```

What are

```
a

b
```

No idea.

With hints

```python
def calculate(price: float,
              tax: float) -> float:
```

Intent is obvious.

# Function Contracts

A contract defines what a function expects and guarantees.

Example

```python
def divide(a: float, b: float) -> float:
```

Contract

Input

- Two numbers

Requirement

```
b != 0
```

Output

A floating-point result.

---

Another example

```python
def average(scores: list[int]) -> float:
```

Requirements

- List should not be empty.

Guarantee

Returns average.

---

Good functions clearly communicate their expectations.

---

# Validate at Boundaries

Never trust external input.

Bad

```python
def divide(a, b):
    return a / b
```

Better

```python
def divide(a, b):
    if b == 0:
        raise ValueError("Division by zero")

    return a / b
```

Validate early.

Fail early.

# Docstrings

A docstring explains

- What the function does
- Parameters
- Return value
- Exceptions

Example

```python
def square(n: int) -> int:
    """
    Returns the square of a number.
    """
    return n * n
```

---

## Good Docstring

```python
def divide(a: float, b: float) -> float:
    """
    Divide two numbers.

    Args:
        a: Numerator.
        b: Denominator.

    Returns:
        Division result.

    Raises:
        ValueError:
            If denominator is zero.
    """
```

---

## Why Docstrings Matter

- Better documentation
- IDE support
- Easier onboarding
- Self-explanatory code

---

# Good API Design

A good function should

- Have one responsibility.
- Have a clear name.
- Validate inputs.
- Return predictable output.
- Avoid unnecessary side effects.

Example

Bad

```python
def process(data):
```

Good

```python
def calculate_total(cart_items):
```

Immediately understandable.

# Common Beginner Mistakes

❌ Using mutable default arguments.

❌ Depending only on comments instead of good function names.

❌ Writing functions without type hints.

❌ Returning different data types depending on conditions.

Example

```python
return 0
```

or

```python
return "Error"
```

Be consistent.

❌ Using `print()` instead of `return`.

❌ Skipping input validation.

---

# LLM Code Review Checklist

Whenever an LLM generates a function, ask:

- Are type hints present?
- Are parameter names meaningful?
- Is the return type consistent?
- Are mutable default arguments avoided?
- Is invalid input validated?
- Is the function contract clear?
- Would another developer understand this function without extra explanation?

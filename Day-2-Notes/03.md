# Day 2 – Functions & Abstraction (Part 3)

# What is Abstraction?

Abstraction means **showing only what is necessary and hiding unnecessary implementation details.**

The caller should know **what** a function does, not **how** it does it.

---

## Example

Instead of writing

```python
marks = [80, 90, 70]

total = 0

for mark in marks:
    total += mark

average = total / len(marks)
```

Create a function

```python
def calculate_average(marks):
    return sum(marks) / len(marks)
```

Now use

```python
average = calculate_average(marks)
```

The caller doesn't care about the loop.

---

## Real-Life Example

You press

```
Power Button
```

on your laptop.

You don't need to know

- CPU initialization
- Memory allocation
- Hardware checks

Those details are hidden.

That's abstraction.

# Levels of Abstraction

Low Level

```python
total = 0

for item in cart:
    total += item.price
```

Even Higher


Higher-level code should read almost like English.

# DRY Principle

**D**on't

**R**epeat

**Y**ourself

---

Bad

```python
math_total = math1 + math2 + math3

science_total = science1 + science2 + science3

english_total = english1 + english2 + english3
```

Repeated logic.

---

Better

```python
def total(scores):
    return sum(scores)
```

Use

```python
math = total(math_scores)

science = total(science_scores)

english = total(english_scores)
```

---

## Benefits

- Less code
- Easier maintenance
- Fewer bugs

# Rule of Three

Don't immediately create a function after writing something once.

General guideline

1st time

Write it normally.

2nd time

Notice repetition.

3rd time

Extract a function.

This avoids creating unnecessary abstractions.

---

# High Cohesion

A function should perform closely related tasks.

Good

```python
def validate_email(email):
```

Everything inside relates to email validation.

---

Bad

```python
def process():
```

Inside it

- Validate email
- Save database
- Send email
- Print report
- Generate invoice

Low cohesion.

---

# Low Coupling

Functions should depend on as few external things as possible.

Bad

```python
database

↓

student

↓

global settings

↓

config

↓

logger

↓

network
```

One function depends on everything.

---

Better

```python

```

Needs only

```
cart
```

Easy to test.

Easy to reuse.

# Function Composition

Small functions work together.

Example

```python
def subtotal(cart):
    ...

def tax(amount):
    ...

def final_price(cart):
    return subtotal(cart) + tax(subtotal(cart))
```

Each function has one responsibility.

Combined, they solve a larger problem.

# First-Class Functions

In Python, functions are objects.

You can

- Store them
- Pass them
- Return them

Example

```python
def greet():
    print("Hello")

x = greet

x()
```

Output

```
Hello
```

Notice

```python
x = greet
```

No parentheses.

You stored the function itself.

# Functions as Arguments

Example

```python
def square(x):
    return x * x

numbers = [1,2,3]

result = map(square, numbers)
```

Another example

```python
sorted(names, key=len)
```

`len` is passed as a function.

# Nested Functions

Functions can exist inside other functions.

```python
def outer():

    def inner():
        print("Hello")

    inner()
```

Useful when helper functions are needed only inside another function.

---

# Closures

A closure remembers variables from its outer scope even after the outer function has finished.

Example

```python
def multiplier(x):

    def multiply(y):
        return x * y

    return multiply
```

Usage

```python
double = multiplier(2)

print(double(5))
```

Output

```
10
```

Why?

The inner function remembers

```
x = 2
```

---

Another Example

```python
add10 = multiplier(10)

print(add10(3))
```

Output

```
30
```

Different returned functions remember different values.

---

# Why Closures?

Useful for

- Configuration
- Function factories
- Decorators
- Callbacks

# Lambda Functions

Small anonymous functions.

Instead of

```python
def square(x):
    return x * x
```

Write

```python
square = lambda x: x * x
```

---

Sorting

```python
students = [
    ("Alice", 90),
    ("Bob", 75)
]

students.sort(key=lambda s: s[1])
```

Sort by marks.

---

When NOT to use lambda

Bad

```python
lambda x:
...
...
...
```

Lambda should stay short.

If logic becomes complex,

use

```python
def
```

# Decorators

A decorator adds functionality to a function without modifying its code.

Think

```
Original Function

↓

Decorator

↓

Enhanced Function
```

---

Simple Example

```python
def logger(func):

    def wrapper():
        print("Before")

        func()

        print("After")

    return wrapper
```

Usage

```python
@logger
def hello():
    print("Hello")
```

Output

```
Before

Hello

After
```

---

Real Uses

- Authentication
- Logging
- Timing
- Caching
- Validation

Frameworks like Flask, Django, and Frappe use decorators extensively.

Example

```python
@app.route("/")
```

or

```python
@frappe.whitelist()
```

These are decorators.

# Higher-Order Functions

A higher-order function

- Accepts another function

or

- Returns another function

Examples

```python
map()

filter()

sorted()

reduce()
```

---

# map()

```python
numbers = [1,2,3]

result = map(lambda x: x*x, numbers)

print(list(result))
```

Output

```
[1,4,9]
```

---

# filter()

```python
numbers = [1,2,3,4]

result = filter(lambda x: x%2==0, numbers)

print(list(result))
```

Output

```
[2,4]
```

---

# Should You Always Use map/filter?

No.

Many Python developers prefer comprehensions.

Instead of

```python
list(map(lambda x:x*x,numbers))
```

Prefer

```python
[x*x for x in numbers]
```

Usually more readable.

# Code Review Exercise

Review

```python
def process(student):

    marks = sum(student)

    avg = marks / len(student)

    print(avg)

    if avg >= 50:
        print("Pass")
    else:
        print("Fail")
```

Problems

- Calculates average.
- Prints.
- Determines result.

Three responsibilities.

---

Better

```python
def average(scores):
    return sum(scores) / len(scores)
```

```python
def result(avg):
    return avg >= 50
```

Caller

```python
avg = average(student)

print(avg)

print(result(avg))
```

Each function has one responsibility.

# Day 2 Cheat Sheet

✔ Functions should be **small, focused, and reusable**.

✔ Prefer **SRP** (Single Responsibility Principle).

✔ Use **descriptive names**.

✔ Return values instead of printing.

✔ Avoid mutable default arguments.

✔ Add **type hints** and **docstrings**.

✔ Validate inputs at boundaries.

✔ Follow **DRY**, but don't over-abstract (Rule of Three).

✔ Prefer **composition** over giant functions.

✔ Functions are **first-class objects**.

✔ Closures remember outer variables.

✔ Decorators extend behavior without modifying code.

✔ Use **list comprehensions** over `map()`/`filter()` when they are more readable.

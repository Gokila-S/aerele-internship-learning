# Why Error Handling Matters

Every real application encounters errors.

Examples

- User enters invalid input.
- Database connection fails.
- File doesn't exist.
- API server is down.
- Network timeout.
- Division by zero.

Good software **expects** these failures and handles them gracefully.

# What is an Exception?

An exception is an event that interrupts the normal flow of a program.

Example

```python
10 / 0
```

Output

```
ZeroDivisionError
```

The program stops because an exception occurred.

---

# Why Use Exceptions?

Without exceptions

```python
result = divide(a, b)

if result == -1:
    print("Error")
```

Question

What if

```
-1
```

is actually a valid answer?

Error codes are ambiguous.

---

Better

```python
def divide(a, b):

    if b == 0:
        raise ValueError("Division by zero")

    return a / b
```

Now

Errors and valid values are separate.

---

# Exceptions vs Error Codes

Error Code

```python
return -1
```

Problems

- Easy to ignore.
- Can conflict with valid values.
- Every caller must check.

---

Exception

```python
raise ValueError(...)
```

Advantages

- Impossible to ignore accidentally.
- Carries useful information.
- Stops incorrect execution.
- Easier debugging.

---

# Raising Exceptions

Use

```python
raise
```

Example

```python
age = -5

    raise ValueError("Age cannot be negative")
```

# try / except

Allows programs to recover from expected failures.

Example

```python
try:
    number = int(input())
except ValueError:
    print("Please enter a number.")
```

Instead of crashing,

the program handles the error.

---

# Execution Flow

```
try

↓

No Error

↓

Continue
```

OR

```
try

↓

Exception

↓

except

↓

Continue
```

---

# Multiple Exceptions

```python
try:
    ...
except ValueError:
    ...
except TypeError:
    ...
```

Handle different failures separately.

---

# Catching Multiple Exceptions

```python
except (TypeError, ValueError):
    ...
```

Useful when the recovery logic is identical.

---

# `else`

Runs only if no exception occurs.

```python
try:
    value = int(input())
except ValueError:
    print("Invalid")
else:
    print("Valid")
```

---

# `finally`

Always executes.

Example

```python
file = open("marks.txt")

try:
    ...
finally:
    file.close()
```

Even if an exception occurs,

the file is closed.

---

Today we usually prefer

```python
with open(...)
```

because it automatically handles cleanup.

---

# Exception Hierarchy

```
BaseException

↓

Exception

↓

ValueError

↓

TypeError

↓

KeyError

↓

...
```

Almost always catch

```
Exception
```

Never catch

```
BaseException
```

because it also catches system-level exceptions like `KeyboardInterrupt`.

---

# Avoid Bare `except`

Bad

```python
try:
    ...
except:
    print("Error")
```

Problem

You don't know what failed.

---

Better

```python
except ValueError:
```

Catch specific exceptions.

---

# Re-raising Exceptions

Sometimes you want to log an error but still let it propagate.

```python
try:
    process()
except ValueError:
    logger.exception("Processing failed")
    raise
```

The exception is preserved.

```python
def divide(a, b):
```

```python

    ...
```

# Interview Questions

### What is an exception?

An event that interrupts normal program execution.

---

### Why are exceptions better than error codes?

- Cannot be silently ignored.
- Carry detailed information.
- Separate errors from valid return values.
- Improve readability.

---

### Difference between `raise` and `except`?

`raise`

Creates an exception.

`except`

Handles an exception.

---

### Difference between `finally` and `else`?

`else`

Runs only if no exception occurred.

`finally`

Always runs.

---

### Why avoid bare `except`?

It hides the actual problem and may catch unexpected exceptions.

---

# Common Beginner Mistakes

❌ Returning `-1` or `"Error"` instead of raising exceptions.

❌ Catching every exception with `except:`.

❌ Swallowing exceptions without logging them.

❌ Using exceptions for normal program flow.

❌ Forgetting cleanup after failures.

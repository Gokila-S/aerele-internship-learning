# Day 5 – Testing, Code Quality & AI Code Review (Part 3)

# Why Testing?

Imagine writing a function.

```python
def add(a, b):
    return a + b
```

Looks correct.

But how do you know it works for every case?

Testing gives confidence that your code behaves correctly.

Professional developers don't assume code works—they verify it.

---

# Types of Testing

## Manual Testing

You run the program yourself.

```
Run

↓

Click Buttons

↓

Observe Output
```

Problems

- Slow
- Repetitive
- Human mistakes
- Doesn't scale

---

## Automated Testing

Tests run automatically.

```
Run Tests

↓

Pass

or

Fail
```

Fast and repeatable.

Preferred in professional software.

---

# Why Pytest?

Python has

```
unittest
```

But most companies prefer

```
pytest
```

because it is

- Simple
- Readable
- Powerful
- Less boilerplate

---

# Installing Pytest

```bash
pip install pytest
```

---

# Writing Your First Test

Suppose

```python
def add(a, b):
    return a + b
```

Test

```python
def test_add():

    assert add(2, 3) == 5
```

Run

```bash
pytest
```

If assertion succeeds

```
PASS
```

Otherwise

```
FAIL
```

---

# Test Naming

Pytest automatically discovers

```
test_*.py
```

Example

```
test_student.py

test_login.py

test_invoice.py
```

Functions

```python
def test_login():
```

must also begin with

```
test_
```

---

# Assertions

Main tool in pytest.

Example

```python
assert square(4) == 16
```

If false

pytest reports

- Expected
- Actual
- Line number

---

# Good Test Cases

Don't test only happy paths.

Example

```python
def divide(a, b):
    ...
```

Test

Normal input

```
10 / 2
```

Boundary

```
10 / 1
```

Invalid input

```
10 / 0
```

Negative values

```
-10 / 2
```

Good tests cover different scenarios.

---

# Testing Exceptions

Suppose

```python
raise ValueError(...)
```

Test

```python
import pytest

def test_invalid_age():

    with pytest.raises(ValueError):
        register(-5)
```

The test passes only if the exception is raised.

---

# Test Structure

Professional tests follow

```
Arrange

↓

Act

↓

Assert
```

Example

```python
def test_add():

    # Arrange
    a = 5
    b = 2

    # Act
    result = add(a, b)

    # Assert
    assert result == 7
```

Simple and readable.

---

# Independent Tests

Each test should work alone.

Bad

```
Test A

↓

Creates student

↓

Test B

Depends on student
```

If Test A fails,

Test B also fails.

Bad design.

Every test should be independent.

---

# Repeatable Tests

Good tests always produce the same result.

Avoid

- Current time
- Random numbers
- Internet dependency
- Real database

unless properly controlled.

---

# Code Quality

Working code isn't always good code.

Good code is

- Readable
- Maintainable
- Testable
- Consistent

---

# Linting

A linter finds problems without running the code.

Examples

- Unused imports
- Unused variables
- Style violations
- Possible bugs

---

# Ruff

Modern Python linter.

Install

```bash
pip install ruff
```

Run

```bash
ruff check .
```

Example

```python
import math

x = 5
```

Output

Unused import

```
math
```

Remove it.

---

# Formatting

Formatting makes code consistent.

Instead of arguing about spaces,

let a formatter decide.

Example

```python
black .
```

or

```bash
ruff format .
```

Benefits

- Consistent style
- Easier reviews
- Cleaner Git diffs

---

# Type Checking

Python is dynamically typed.

Sometimes mistakes are discovered only at runtime.

Example

```python
def add(a: int, b: int):
    return a + b

add("A", "B")
```

A type checker warns before execution.

---

# MyPy

Install

```bash
pip install mypy
```

Run

```bash
mypy .
```

It checks type hints for possible errors.

---

# Pre-commit Hooks

A hook runs automatically before every Git commit.

Example

```
git commit

↓

Run Ruff

↓

Run Formatter

↓

Run Tests

↓

Commit
```

If checks fail,

the commit is blocked.

This keeps bad code out of the repository.

---

# Why Use Pre-commit?

Instead of remembering

```
Run formatter

Run linter

Run tests
```

Git does it automatically.

---

# AI Code Review

Modern developers use LLMs to review code.

But don't blindly accept suggestions.

Treat AI as a junior reviewer.

---

# Questions to Ask AI

Instead of

```
Is this code good?
```

Ask

```
Can this function be simplified?

Are there edge cases?

Any security issues?

Can naming improve?

Any hidden bugs?

Any unnecessary complexity?
```

Specific prompts produce better reviews.

---

# Verify AI Suggestions

Never merge code immediately.

Check

- Does it still pass tests?
- Is it more readable?
- Did functionality change?
- Is performance affected?
- Is the suggestion actually correct?

AI can make mistakes.

---

# Code Review Checklist

Whenever reviewing code, ask

- Is the function too long?
- Are variable names meaningful?
- Are responsibilities mixed?
- Can duplication be removed?
- Is error handling correct?
- Are edge cases covered?
- Are tests available?
- Is the code readable?

---

# Code Review Example

Review

```python
def login(u, p):
```

Problem

Poor variable names.

Better

```python
def login(username, password):
```

---

Review

```python
def calculate():

    print("Start")

    ...

    print("Done")
```

Better

```python
logger.info("Calculation started")
```

---

Review

Huge function

```
350 lines
```

Better

Split into

```
validate()

calculate()

save()

notify()
```

Each function should have one responsibility.

# Interview Questions

### Why use pytest?

Because it is simple, readable, and widely used for automated testing.

---

### What does `assert` do?

Checks whether an expected condition is true.

If not, the test fails.

---

### Why use Ruff?

To detect style issues, unused code, and potential bugs.

---

### Why use MyPy?

To find type-related mistakes before running the program.

---

### What is a pre-commit hook?

A script that automatically runs checks before Git allows a commit.

---

### Should AI-generated code be trusted blindly?

No.

Always review, test, and verify every suggestion.

---

# Common Beginner Mistakes

❌ Testing only one input.

❌ Depending on other tests.

❌ Ignoring linting errors.

❌ Not formatting code.

❌ Skipping tests because "it works on my machine."

❌ Accepting AI suggestions without verification.

---

# LLM Code Review Checklist

Whenever ChatGPT generates code, ask:

- Does it follow the Single Responsibility Principle?
- Are edge cases tested?
- Can naming improve?
- Is exception handling correct?
- Is duplication present?
- Would Ruff report any issues?
- Would MyPy report any type errors?
- Are tests sufficient before merging?

# Day 5 Cheat Sheet

✔ Use **exceptions**, not error codes.

✔ Validate input at application boundaries.

✔ Fail fast with meaningful exceptions.

✔ Use **logging**, not `print()`.

✔ Prefer `logger.exception()` when handling unexpected errors.

✔ Write automated tests with **pytest**.

✔ Test happy paths, edge cases, and failures.

✔ Use **Ruff** for linting and formatting.

✔ Use **MyPy** for type checking.

✔ Automate quality checks with **pre-commit**.

✔ Treat AI as a reviewer—not as the final authority.

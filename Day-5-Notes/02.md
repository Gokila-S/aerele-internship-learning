# Day 5 – Exceptions, Logging & Production Quality (Part 2)

# Custom Exceptions

Built-in exceptions are useful, but sometimes your application needs errors that describe business rules.

Instead of

```python
raise ValueError("Balance")
```

Create your own exception.

```python
class InsufficientBalanceError(Exception):
    pass
```

Usage

```python
if balance < amount:
    raise InsufficientBalanceError(
        "Insufficient account balance"
    )
```

Now the error clearly describes the problem.

---

## Why Custom Exceptions?

Imagine an ERP system.

Possible errors

```
EmployeeNotFoundError

InvoiceAlreadyPaidError

InvalidAttendanceError

InsufficientStockError

CourseFullError
```

These names immediately explain what happened.

---

# Exception Inheritance

Custom exceptions usually inherit from

```python
Exception
```

Example

```python
class StudentError(Exception):
    pass
```

More specific exceptions

```python
class InvalidAgeError(StudentError):
    pass
```

```python
class DuplicateRollNumberError(StudentError):
    pass
```

Hierarchy

```
Exception

↓

StudentError

↓

InvalidAgeError

↓

DuplicateRollNumberError
```

Now

```python
except StudentError:
```

can handle every student-related exception.

# Validate at Boundaries

One of the most important software engineering principles.

Never trust external input.

External input includes

- User input
- API requests
- Files
- Database
- Forms
- Browser requests

Validate immediately.

---

Bad

```python
def register(age):
    return age + 10
```

If

```
age = "abc"
```

Program crashes later.

---

Better

```python
def register(age):

    if not isinstance(age, int):
        raise TypeError(
            "Age must be an integer."
        )

    if age < 0:
        raise ValueError(
            "Age cannot be negative."
        )

    return age
```

Fail early.

---

# Fail Fast Principle

Detect problems immediately.

Bad

```
Invalid data

↓

Many functions

↓

Crash
```

Hard to debug.

---

Better

```
Invalid data

↓

Validation

↓

Immediate Error
```

Easy to locate.

# Defensive Programming

Never assume

```
"It will never happen."
```

Instead ask

```
"What if it does?"
```

Examples

Check

- None
- Empty lists
- Invalid IDs
- Missing files
- Database failures
- Network timeout

Professional software assumes failures are possible.

# Assertions

Assertions verify assumptions during development.

Example

```python
def divide(a, b):

    assert b != 0

    return a / b
```

If

```
b == 0
```

Python raises

```
AssertionError
```

---

## When NOT to Use Assertions

Don't validate user input with assertions.

Wrong

```python
assert age >= 18
```

Users can trigger this.

Use

```python
raise ValueError(...)
```

Assertions are mainly for detecting programmer mistakes during development.

# Logging

One of the biggest differences between beginner and production code.

Beginners

```python
print("User Logged In")
```

Professionals

```python
import logging

logger = logging.getLogger(__name__)

logger.info("User Logged In")
```

---

# Why Logging?

Logs can be

- Saved to files
- Filtered
- Searched
- Sent to monitoring systems

Print statements disappear after execution.

Logs remain.

---

# Log Levels

Python provides different logging levels.

```
DEBUG
```

Detailed debugging information.

---

```
INFO
```

Normal application events.

Example

```
User logged in.
```

---

```
WARNING
```

Unexpected but recoverable.

Example

```
Disk space running low.
```

---

```
ERROR
```

An operation failed.

Application may continue.

---

```
CRITICAL
```

Severe error.

Application may stop.

---

# Logging Example

```python

logger.info("Student created")

```

Choose the correct level.

# logger.error() vs logger.exception()

Very common interview question.

---

## logger.error()

Logs an error message only.

```python
except ValueError:

    logger.error(
        "Invalid age"
    )
```

Output

```
ERROR

Invalid age
```

No stack trace.

---

## logger.exception()

Must be called inside

```
except
```

Example

```python
try:
    process()

except Exception:

    logger.exception(
        "Processing failed"
    )
```

Output

```
Processing failed

Full traceback...
```

Stack trace included.

---

## When to Use Which?

Need only a message?

Use

```python
logger.error()
```

Need traceback?

Use

```python
logger.exception()
```

---

# Don't Ignore Exceptions

Bad

```python
try:
    process()

except Exception:
    pass
```

The error disappears.

Debugging becomes impossible.

---

Better

```python
except Exception:

    logger.exception(
        "Processing failed"
    )

    raise
```

Now

- Error is logged.
- Caller still knows something failed.

---

# Wrapping Exceptions

Sometimes lower-level exceptions are not meaningful.

Example

```python
try:
    db.save()
except DatabaseError:
    raise StudentSaveError(
        "Unable to save student."
    )
```

Caller now receives a business-specific exception.

# Interview Questions

### What is a custom exception?

An application-specific exception created by inheriting from `Exception`.

---

### Why create custom exceptions?

To represent meaningful business errors instead of generic ones.

---

### What does "Validate at Boundaries" mean?

Validate external input as soon as it enters the application.

---

### Difference between `logger.error()` and `logger.exception()`?

`logger.error()`

Logs only the message.

`logger.exception()`

Logs the message **and** the complete traceback.

---

### Should we ever write

```python
except:
```

No.

Always catch specific exceptions whenever possible.

---

### Should exceptions ever be ignored?

No.

At minimum,

- Log them.
- Handle them.
- Or re-raise them.

# Common Beginner Mistakes

❌ Using `print()` instead of logging.

❌ Catching every exception.

❌ Swallowing exceptions with `pass`.

❌ Using generic `Exception`.

❌ Validating user input with `assert`.

❌ Delaying validation until deep inside the application.

- Is the exception type specific?

- Should `logger.exception()` be used instead of `logger.error()`?

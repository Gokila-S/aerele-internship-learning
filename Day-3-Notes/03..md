# Day 3 – Object-Oriented Programming (Part 3)

# Dataclasses

One of the biggest improvements in modern Python.

If a class mainly stores data, writing constructors and utility methods manually becomes repetitive.

Example without dataclass

```python
class Student:

    def __init__(self, name, age, dept):
        self.name = name
        self.age = age
        self.dept = dept
```

Lots of boilerplate.

---

## Using Dataclass

```python
from dataclasses import dataclass

@dataclass
class Student:
    name: str
    age: int
    dept: str
```

Python automatically generates

- `__init__()`
- `__repr__()`
- `__eq__()`

No extra code needed.

---

## Usage

```python
student = Student("Alice", 20, "CSE")

print(student)
```

Output

```
Student(name='Alice', age=20, dept='CSE')
```

---

## Why Use Dataclasses?

Perfect when the class only stores data.

Examples

- Student
- Employee
- Invoice
- Product
- Order

---

## Mutable Default Values

Avoid

```python
@dataclass
class Student:
    marks: list = []
```

Same problem as function defaults.

Correct

```python
from dataclasses import field

@dataclass
class Student:
    marks: list = field(default_factory=list)
```

Each object gets a new list.

# NamedTuple

Before dataclasses, Python used `namedtuple` for lightweight data containers.

```python
from collections import namedtuple

Student = namedtuple(
    "Student",
    ["name", "age"]
)

student = Student("Alice", 20)

print(student.name)
```

---

## Difference

NamedTuple

- Immutable
- Lightweight
- Faster

Dataclass

- Mutable by default
- Easier to extend
- More common today

---

# Enum

An Enum represents a fixed set of constant values.

Without Enum

```python
status = "OPEN"
```

What if someone writes

```python
status = "OPEEN"
```

Bug.

---

Using Enum

```python
from enum import Enum

class Status(Enum):

    OPEN = "OPEN"

    CLOSED = "CLOSED"

    PENDING = "PENDING"
```

Usage

```python
status = Status.OPEN
```

Now invalid values aren't accidentally created.

---

## Why Enums?

- Prevent magic strings
- Improve readability
- Better autocomplete
- Fewer bugs

# Dunder (Magic) Methods

"Dunder" means

```
Double Underscore
```

Example

```
__init__

__str__

__repr__

__len__

__eq__
```

Python automatically calls these methods.

---

# `__str__`

Human-readable representation.

```python
class Student:

    def __str__(self):
        return self.name
```

```python
print(student)
```

Output

```
Alice
```

---

# `__repr__`

Developer-friendly representation.

```python
def __repr__(self):
    return f"Student('{self.name}')"
```

Useful while debugging.

---

# `__len__`

```python
class Team:

    def __len__(self):
        return len(self.members)
```

Now

```python
len(team)
```

works.

---

# `__eq__`

Controls equality.

```python
class Student:

    def __eq__(self, other):
        return self.roll_no == other.roll_no
```

Now

```python
student1 == student2
```

uses your logic.

# SOLID Principles

Five principles for maintainable OOP.

Don't memorize the names.

Understand the ideas.

---

# S — Single Responsibility Principle

A class should have one reason to change.

Bad

```python
class Student:

    save_database()

    send_email()

    print_report()

    calculate_grade()
```

Too many jobs.

Better

```
Student

↓

StudentRepository

↓

EmailService

↓

ReportGenerator
```

---

# O — Open/Closed Principle

Classes should be

```
Open for Extension

Closed for Modification
```

Instead of changing existing code,

extend it.

Example

Add a new payment method by creating another class instead of editing every existing payment class.

---

# L — Liskov Substitution Principle

Child objects should replace parent objects without breaking code.

Example

Good

```
Animal

↓

Dog
```

Every Dog behaves like an Animal.

Bad

```
Bird

↓

Penguin
```

If Bird assumes

```
fly()
```

Penguin breaks that assumption.

Inheritance design is wrong.

---

# I — Interface Segregation Principle

Don't force classes to implement methods they don't need.

Instead of

```python
class Worker:

    code()

    test()

    design()

    deploy()
```

Split responsibilities into smaller interfaces.

Python often achieves this through duck typing instead of explicit interfaces.

---

# D — Dependency Inversion Principle

Depend on abstractions,

not concrete implementations.

Bad

```python
class Order:

    payment = Razorpay()
```

Hard to change.

Better

```python
class Order:

    def __init__(self, payment):
        self.payment = payment
```

Now

```
Stripe

Razorpay

PayPal
```

all work.

# Dataclass vs Normal Class

Use Dataclass when

- Mostly storing data
- Minimal behavior

Use Normal Class when

- Complex logic
- Multiple methods
- Validation
- State changes

# Interview Questions

### What is a dataclass?

A decorator that automatically generates boilerplate methods like `__init__`, `__repr__`, and `__eq__` for data-centric classes.

---

### Dataclass vs NamedTuple?

Dataclass

- Mutable
- Rich features
- Preferred today

NamedTuple

- Immutable
- Lightweight

---

### What is an Enum?

A fixed collection of named constants.

---

### What are dunder methods?

Special methods Python calls automatically to customize object behavior.

---

### Difference between `__str__` and `__repr__`?

`__str__`

Human-readable output.

`__repr__`

Developer/debug representation.

---

### Which SOLID principle is most important?

All are useful, but **Single Responsibility Principle (SRP)** is the one you'll apply most often in everyday development.

---

# Common Beginner Mistakes

❌ Writing boilerplate classes instead of using `@dataclass`.

❌ Using strings instead of Enums for fixed values.

❌ Ignoring special methods like `__str__()`.

❌ Creating classes with too many responsibilities.

❌ Treating inheritance as the default solution.

❌ Forgetting that composition is usually simpler.

---

# LLM Code Review Checklist

Whenever ChatGPT generates classes, ask:

- Could this be a `@dataclass`?
- Is an Enum better than magic strings?
- Are relationships modeled correctly?
- Does each class have one responsibility?
- Are dunder methods needed for readability or comparison?
- Is dependency injection possible instead of hardcoding dependencies?

# Day 3 Cheat Sheet

✔ A **class** is a blueprint.

✔ An **object** is an instance of a class.

✔ Use `self` to refer to the current object.

✔ Use `__init__()` to initialize objects.

✔ Prefer **composition** over inheritance when it's a "has-a" relationship.

✔ Use `super()` to reuse parent initialization.

✔ `@dataclass` removes boilerplate for data models.

✔ Use **Enums** for fixed sets of values.

✔ Dunder methods customize built-in behavior.

✔ Follow **SOLID**, especially SRP.

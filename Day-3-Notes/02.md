# Day 3 – Object-Oriented Programming (Part 2)

# Inheritance

Inheritance allows one class to reuse the properties and methods of another class.

Instead of rewriting common code, inherit it.

---

## Why Use Inheritance?

Suppose you're building an ERP system.

Without inheritance

```python
class Employee:
    name
    email

class Manager:
    name
    email

class Intern:
    name
    email
```

The same attributes are repeated.

Better

```python
class Employee:

    def __init__(self, name, email):
        self.name = name
        self.email = email
```

```python
class Manager(Employee):
    pass
```

```python
class Intern(Employee):
    pass
```

Now both reuse Employee.

# Example

```python
class Animal:

    def speak(self):
        print("Animal Sound")
```

```python
class Dog(Animal):
    pass
```

Usage

```python
dog = Dog()

dog.speak()
```

Output

```
Animal Sound
```

Dog inherited the method.

# Method Overriding

Child classes can replace parent behavior.

```python
class Animal:

    def speak(self):
        print("Animal")
```

```python
class Dog(Animal):

    def speak(self):
        print("Bark")
```

Usage

```python
Dog().speak()
```

Output

```
Bark
```

Python chooses the child version.

---

# Calling Parent Methods (`super()`)

Sometimes we want to extend the parent behavior instead of replacing it.

```python
class Employee:

    def __init__(self, name):
        self.name = name
```

```python
class Manager(Employee):

    def __init__(self, name, team):

        super().__init__(name)

        self.team = team
```

`super()` calls the parent constructor.

---

# Why Use `super()`?

Without it

```python
self.name = name
```

must be rewritten.

With it

```python
super().__init__(name)
```

Parent initialization is reused.

# Method Resolution Order (MRO)

If multiple inheritance exists,

Python needs to know

Which method should run?

Example

```python
class A:

    def show(self):
        print("A")
```

```python
class B(A):
    pass
```

```python
class C(A):
    pass
```

```python
class D(B, C):
    pass
```

Python follows

```
D

↓

B

↓

C

↓

A
```

Check using

```python
D.mro()
```

# Polymorphism

Same interface.

Different behavior.

Example

```python
class Dog:

    def speak(self):
        print("Bark")
```

```python
class Cat:

    def speak(self):
        print("Meow")
```

Function

```python
def make_sound(animal):
    animal.speak()
```

Works for both.

```
Dog

↓

Bark

Cat

↓

Meow
```

No change required.

---

# Why Polymorphism?

Instead of

```python
if animal == Dog:

...

elif animal == Cat:

...
```

Simply call

```python
animal.speak()
```

Cleaner.

More extensible.

---

# Duck Typing vs Inheritance

Many beginners think inheritance is required.

Not true.

Duck typing

```python
class Robot:

    def speak(self):
        print("Beep")
```

Still works

```python
make_sound(Robot())
```

Even though

```
Robot
```

doesn't inherit

```
Animal
```

Python cares about behavior.

# Composition (Preferred in Python)

Composition means

A class contains another class.

Instead of

```
IS-A
```

relationship,

use

```
HAS-A
```

relationship.

---

Example

Instead of

```python
class Engine:
```

```python
class Car(Engine):
```

Wrong.

A car

is not

an engine.

Correct

```python
class Engine:
    ...
```

```python
class Car:

    def __init__(self):

        self.engine = Engine()
```

Car **has an** Engine.

---

# Why Composition?

Inheritance creates tight coupling.

Composition creates flexibility.

Suppose

```
Petrol Engine

↓

Electric Engine
```

With composition,

changing engines is easy.

---

# Composition Example

```python
class Address:

    def __init__(self, city):
        self.city = city
```

```python
class Student:

    def __init__(self, name, address):

        self.name = name

        self.address = address
```

Usage

```python
address = Address("Erode")

student = Student("Alice", address)
```

Access

```python
student.address.city
```

# When to Use Inheritance?

Ask

Is it truly an

```
IS-A
```

relationship?

```
Manager IS-A Employee

Dog IS-A Animal
```

Good.

---

When to Use Composition?

Ask

Does it have a

```
HAS-A
```

relationship?

```
Car HAS-A Engine

Student HAS-A Address

Company HAS-A Employees
```

Prefer composition whenever possible.

# Interview Questions

### What is inheritance?

Reusing properties and methods from another class.

---

### What is method overriding?

Replacing a parent method in the child class.

---

### What is `super()`?

Calls the parent implementation.

---

### What is polymorphism?

Same interface.

Different implementations.

---

### Difference between Inheritance and Composition?

Inheritance

```
IS-A
```

Composition

```
HAS-A
```

---

### Which is preferred in Python?

Composition.

Use inheritance only for genuine "is-a" relationships.

---

### What is MRO?

Method Resolution Order.

Determines which method Python calls first in multiple inheritance.

---

# Common Beginner Mistakes

❌ Using inheritance everywhere.

❌ Forgetting `super()`.

❌ Creating deep inheritance chains.

❌ Using `isinstance()` repeatedly instead of polymorphism.

❌ Confusing "has-a" with "is-a".

---

# LLM Code Review Checklist

Whenever an LLM generates classes, ask:

- Is inheritance actually required?
- Would composition make the design simpler?
- Is `super()` used correctly?
- Is overriding necessary?
- Is polymorphism replacing long `if/elif` chains?
- Are relationships modeled correctly (`IS-A` vs `HAS-A`)?

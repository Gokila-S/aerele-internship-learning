# Topic 2 — Names are References, Not Boxes

# Why does this concept exist?

Most programming languages teach variables like this.

```
Variable
↓

Stores

↓

Value
```

For example

```
x = 10
```

We imagine

```
┌──────┐
│  x   │
├──────┤
│  10  │
└──────┘
```

This picture is **wrong** in Python.

Python variables do **not store values.**

They store **references** to objects.

That small difference changes everything.

---

# Mental Model

Imagine memory as a large city.

Objects are houses.

Variables are simply address labels.

Example

```python
name = "Python"
```

Python creates the string object.

```
Memory

+----------------+
|   "Python"     |
+----------------+
```

Now the variable

```
name
```

doesn't contain the string.

Instead

```
name
    │
    ▼

+----------------+
|   "Python"     |
+----------------+
```

The variable simply points to it.

Think of variables as **sticky notes**.

The sticky note tells you where the object lives.

It is **not** the object.

---

# Example

```python
a = 10
b = a
```

What happens?

Many beginners think

```
10 copied into b
```

Python actually does

```
      a
      │
      ▼

+---------+
|   10    |
+---------+
      ▲
      │
      b
```

Both variables point to the same object.

---

# Let's Verify

```python
a = 10
b = a

print(id(a))
print(id(b))
```

Output

```
140733...
140733...
```

Same ID.

Same object.

---

# What is id()?

Every object has an identity.

Think of it as its home address.

```python
name = "Python"

print(id(name))
```

returns the object's identity.

You usually don't need `id()` in production.

But it helps understand Python's memory model.

---

# Rebinding

Suppose

```python
a = 10
b = a

a = 20
```

Many beginners think

> Changing `a` changed the object.

That's **not** what happened.

Python created a new object.

```
Initially

        a
        │
        ▼
      +------+
      | 10   |
      +------+
        ▲
        │
        b

```

After

```python
a = 20
```

```
      a
      │
      ▼
   +------+
   | 20   |
   +------+

      b
      │
      ▼
   +------+
   | 10   |
   +------+
```

Notice

The variable moved.

The object didn't.

Variables can change which object they reference.

Objects don't magically change because the variable changed.

---

# Common Misunderstanding

People say

> Variable changed.

Not exactly.

More accurately,

> The variable now refers to a different object.

Python calls this **rebinding**.

---

# Why is this important?

Imagine

```python
user = fetch_user()
```

Later

```python
user = None
```

Did you delete the user?

No.

You only changed what

```
user
```

points to.

---

# Interview Question

Q:

What happens internally when you write

```python
a = b
```

Good answer:

Python doesn't immediately duplicate the object.

It copies the reference.

Both variables point to the same object until one variable is rebound.

# Topic 3 — Mutable vs Immutable Objects

Now that we know variables are references,

we can answer the next question.

Can an object itself change?

Sometimes yes.

Sometimes no.

---

# Why does Python have two kinds of objects?

Imagine a passport.

Once issued,

you don't erase your birth date.

You replace the passport.

That's an immutable object.

Now imagine a notebook.

You erase,

write,

draw,

remove pages.

That's mutable.

Python has both.

---

# Immutable Objects

Once created,

they cannot change.

Examples

```python
int
float
bool
str
tuple
frozenset
```

---

Example

```python
name = "Python"

name += "3"
```

Did Python modify

```
"Python"
```

No.

It created

```
"Python3"
```

and moved

```
name
```

to point at the new object.

```
Old Object

+-----------+
| Python    |
+-----------+

New Object

+-----------+
| Python3   |
+-----------+
```

---

# Mutable Objects

These objects can change after creation.

Examples

```python
list

dict

set
```

Example

```python
numbers = [1,2,3]

numbers.append(4)
```

Did Python create another list?

No.

The same object changed.

Before

```
[1,2,3]
```

After

```
[1,2,3,4]
```

Same object.

Different contents.

---

# Visualizing It

```
numbers
    │
    ▼

+-----------------+
| 1 | 2 | 3 | 4   |
+-----------------+
```

Appending changes the existing object.

The reference remains the same.

---

# Why Lists are Mutable

Imagine if every append created a new list.

```
append()

↓

new list

↓

append()

↓

new list

↓

append()

↓

new list
```

Memory usage would explode.

Lists are designed to be edited.

Strings are designed to stay fixed.

---

# How to Remember

Immutable

> Replace me.

Mutable

> Modify me.

# Topic 4 — Shared References

This is where beginners get surprised.

Consider

```python
a = [1,2,3]

b = a
```

Question

How many lists exist?

Only one.

```
      a
      │
      ▼

+-------------+
|1|2|3|
+-------------+
      ▲
      │
      b
```

Now

```python
b.append(4)
```

What prints?

```python
print(a)
```

Output

```
[1,2,3,4]
```

Many beginners expected

```
[1,2,3]
```

But both variables reference the same object.

Changing the object changes what both variables see.

---

# Why does this happen?

Remember.

Variables don't own objects.

They only know where the object lives.

If two people have the same house address,

both visit the same house.

Changing the furniture changes the experience for everyone.

---

# Common Beginner Mistake

```python
a = [1,2]

b = a

b.append(3)
```

Then asking

> Why did `a` change?

It didn't.

The list changed.

Both variables point to that list.

---

# Senior Developer Perspective

Whenever you pass a list or dictionary to another function,

always ask:

> Is this function allowed to modify it?

If yes,

fine.

If no,

make a copy.

Understanding shared references prevents many real-world bugs.

---

# Key Takeaways

✅ Variables store references, not values.

✅ Objects live independently in memory.

✅ Reassigning a variable changes the reference, not the object.

✅ Immutable objects cannot change.

✅ Mutable objects can change.

✅ Two variables may reference the same mutable object.
